Waterfall to DevOps

September 12th, 2017

By Didier BERNAUDEAU

- Il est très difficile d’apporter une défition au terme « DevOps ».

- C’est un concept relativement simple qui consiste à rapprocher les développeurs et les opérationnels.

- Cependant, DevOps à des impacts non négligeable pour une DSI : organisation, architecture, nouvelles technologies et bien sûre pour la sécurité.

- Donc pour cette première présentation sur le sujet, je vous propose d’aborder la gestion de projet « Waterfall to DevOps ».

Waterfall

The “traditional” approach


linear and sequential (non-iterative)

Cette méthode traditionnelle date des années 1970

C’est une méthode séquentiel et linéaire.

La sécurité s’est intégrée au fils des années.

Cette méthode est relativement contreversée car le taux d’insatisfaction est important. Le client ne verra le résultant qu'à la fin du processus.

Agile

The iterative approach

  • Manifesto for Agile Software Development (February 2001)

  • Popular framework: Scrum and Kanban

  • Product oriented

  • Cross functional team:
    • Small team (Two-pizza team)
    • Self-organizing and self-managing

- La méthode agile à fait sont apparition officiellement en 2001

- Il existe de nombreux framework pour l’implémenter : Scrum, kanban, …

- Changement de paradigme: C’est une méthode orienté produit (et pas projet)

- L’équipe est autonome

- Maximum 12 personnes

Scrum sprint lifecycle

- Le Product Owner définie les users stories dans le product backlog

- Le sprint commence avec le sprint planning pour définir les taches à effectuer dans le Sprint Backlog.

- Le sprint abouti à une évolution du produit potentiellement livrable en production.

- Le processus est itérative avec des sprints durant 2 à 3 semaines.

- Le produit sera livré en production lors de la release après plusieurs sprints.

Agile Security

  • Secure Scrum

  • Product backlog:
    • Security-focused stories (Safe Code)
    • Evil user stories (OWASP) or Abuser stories

  • Sprint backlog: security features will need to be included as applicable for every single sprint backlog feature.

  • Definition of Done: include secure acceptance criteria (Source code audit, ...).

- La méthode agile à un défaut majeur puisque la sécurité n’est pas abordée par le manifesto d’origine.

- "Secure Scrum" est une extension à Scrum permettant de traiter brièvement la sécurité.

- L’intégration de la sécurité dans scrum se fait à 3 niveaux:

  • Product backlog
  • Sprint backlog
  • Definition of done (Chaque user story doit avoir ses critères de sécurité)

Example of user stories


  • As an employee, I can search for other employees by their last name.

  • As an architect/developer I want to ensure AND as QA I want to verify the user is protected by robust authentication and session management.

  • As a hacker, I can send bad data in the content of requests, so I can access data and functions for which I'm not authorized.

Agile Security


  • Prevent "cowboy" development:

    • Select framework and write secure guidelines
    • Select development tools and secure its

- Agile et DevOps ont parfois une mauvaise réputation à cause des dérives : cowboy developement.

- Pour y remédier, la DSI doit définir les framework à utiliser accompagné de ses guides d’utilisation.

- Il faut également définir les outils qui seront mis à disposition des développeurs et il faut évidement les sécuriser.

Agile to DevOps


  • Continous Deployment

  • Organization:
    • Close-knit collaboration between Dev and Ops
    • Dev and Ops in one team

- La principale évolution est la livraison continu en production.

- A l'origine, la méthode agile impose un planning de release pour livrer en production

- En terme d’organisation, il y a 2 options :

  • Collaboration : coopération très forte avec les ops (via le scrum master)
  • Fusion : les ops rejoignent l’équipe de dev

DevOps tools chain

NoOps

The end of DevOps


  • Serverless architecture
  • CloudNative
  • Agile infrastructure
  • IAC (Infrastructure As Code)

- Désormais, l’infrastructure est automatisée à tel point que l’équipe opérationnelle n’est plus réellement nécessaire pour une application.

- Les développeurs peuvent gérer l’infrastructure par de simples API, sans avoir besoin de compétences spécifique.
