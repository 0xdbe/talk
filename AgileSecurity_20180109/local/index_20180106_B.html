Agile Security

By Didier BERNAUDEAU
OSSIR (January 9th, 2018)

Je vous propose de vous présenter comment intégrer la sécurité dans une démarche Agile. La première partie sera consacrée à la méthode agile puis la seconde partie évoquera rapidement tout ce que cela implique (software development pipeline, micro service, container, ...). Il reste 10 a 15 minutes en fin de présentation pour vos questions.

    Product Thinking


    No Project


    No Application

    • Il y a 2 approches possibles : projet ou produit
    • Il y a de nombreux débats sur l'approche à retenir mais la suite de la présentation se fera avec une approche produit
    • L'approche projet est une vision éphémère.
    • L'approche produit impacte la gestion budgétaire et la définition des objectifs des collaborateurs

    • Tout produit contribue à une fonction métier (contrairement à certaine application dont la fonction métier est indirecte)

    Product Backlog


    • List of desired features, bug fixes, non-functional requirements, ...
    • The Product Owner prioritizes the backlog
    • Product Owners have a lot of stakeholders to take into account
      • Ordered list of item :
        • User Stories
      • Only Product Owner can add, modify or delete a user story

      • Acceptance Criteria
        • List of items, each one used to validate a User Story
        • Applicable to all items in the Product Backlog, not just a single User Story.

      • User Stories
      • Security-focused stories (Safe Code)
      • Evil user Storie / Abuser Stories

    Definition of Done

    • List of activities
    • List of items to validate a User Story
    • Applicable to all items in the Product Backlog, not just a single User Story.

    Example of secure activity:
    There should be no open critical and high vulnerability identified by Source Code Analysis
