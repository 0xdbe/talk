Anti-DDOS VAC (1/2)

Par Octave Klaba (OVH)


  • VAC (Vacuum)
    • Mesure de protection réalisée par OVH
    • Solution basée sur des FPGA
    • Protection contre le D-DOS uni-directionel
    • Gratuit pour tous les clients (Cela permet à OVH de protéger son infrastructure)
    • 2,5 Tbps de capacité de nettoyage

  • Détection et activation:
    • La détection de l'attaque DDOS doit se faire le plus rapidement possible
    • Détection par analyse de log (Kafka et développement interne)
    • L'activation consiste à re-router automatiquement les flux vers le VAC
    • L'activation doit se faire en moins d'une seconde

Anti-DDOS VAC (1/2)

Par Octave Klaba (OVH)


  • Multi-VAC:
    • Le flux est routé vers le VAC du datacenter le plus proche de l'origine du flux
    • Après nettoyage, le flux est routé vers le datacenter de destination

  • Technologies:
    • QUAGGA: logiciel de routage open source installé sur les VAC
    • Protocole de routage: DSCP + PBR + BGP + QPPB
    • PBR: Policies Based Routing
    • UBRL (User Based Rate Limiting):
      • Offre un accès équitable à chaque trader
      • Equitable = même temps de réponse pour tous quelque soit le lieu du trader
    • Arbor: solution matériel de nettoyage des flux réseau (plus performant que Radware mais relativement cher)

L'administration en Silo

Par Aurélien Bordes

  • Scénario de test d'intrusion classique :
    • Extraction des secrets kerberos en mémoire (Mimikatz)
    • poste de travail > Serveur > Active directory

    Quelques solutions pour s'en prémunir :

  • Niveau d'administration à mettre en oeuvre:
    • Rouge: Active Directory avec administrateur de domaine
    • Jaune: serveurs avec administrateur des serveurs
    • Vert: poste de travail avec administrateur local et utilisateur du helpdesk

  • Nouveaux mécanismes de sécurité mis en oeuvre par Microsoft :
    • Protected User Security Group permet de renforcer le niveau de sécurité d'un compte utilisateur comme désactiver la délégation de compte et l'utilisation du protocole NTLM
    • FAST (Flexible authentification via Secure Tunneling) est un protocole pour protéger les tickets TGT
    • Stratégie d'authentification permet de mettre en place une restriction sur l'utilisation des tickets Kerberos (exemple: le poste de travail doit être dans le groupe "admin_Ad")
    • Authentification Composée permet de protéger les messages TGS

WSUS pendu

Par Yves LE PROVOST (ANSSI) et Romain COLTEL (ALSID)

Utilisation de WSUS pour compromettre les machines qui se connectent au serveur WSUS d'entreprise

  • Vulnérabilité:
    • les machines se connectent au serveur WSUS en HTTP
    • Le processus s'exécute avec les droits système sur les machines
    • Seul le binaire est signé (pas les paramètres)

  • Attaque WSUSpect (Black Hat USA 2015) :
    • MITM entre la machine et le serveur WSUS
    • Injection de code malveillant en paramètre d'un binaire signé par Microsoft

  • Attaque WSUSpendu:
    • Accès à la base de données du WSUS pour modifier les paramètres
    • Code pour créer un administrateur sur les machines victimes

  • Méthode de protection:
    • Activer le TLS sur le serveur WSUS
    • Le serveur WSUS doit être dans la zone rouge pour limiter les accès illicites à la base de données

Durcissement système à l'aide de Systemd

Par Timotthée Ravier (ANSSI)


  • Systemd est un SysInit qui permet de lancer les autres démons d'un système Linux.

  • Pour éditer la configuration : systemctl edit php-fpm.service

  • Méthodes de sécurisation disponibles:
    • Ajouter les attributs "User" et "Group"
    • Utilisation de seccomp-bpf avec l'attribut "SystemCallFilter"
    • Ajouter des capacités à un processus avec l'attribut "CapabilityBoundingSet" et "Ambient Capabilities"
    • Utilisation des namespaces avec l'option "InaccessiblePaths" et "ProtectSystem"

  • Exemple de protection contre Dirty Cow (Copy on Write)

  • Troll: les vulnérabilités de systemd n'ont pas été abordées !!!

Désobfuscation de binaire

par J. SALWAN, M. POTET, S. BARDIN (CEA et QuaskSlab)


  • Binary Protection:
    • byte code propriétaire (Instruction Set Architecture) émulé par une machine virtuelle
    • Dans certains cas, plusieurs niveau de VM

  • Approche classique: analyse du code dans la machine virtuelle pour analyser le code
  • Approche proposée: automatic désobfuscation
    • Isoler les instructions du programme des instructions de la machine virtuelle (Taint analysis)
    • Reconstruire le control flow du programme en prenant en compte l'état initial de la VM

  • Processus : Obfuscated binary > Triton AST > Arybo > LLVM > Deobfuscated Binary

  • Exemple avec Tigress Challenges

TLS 1.3

Par Filippo VALSORDA (Cloudflare)


  • TLS 1.2 and TLS 1.3 overview

  • RTT (Round Trip Time) Resumption
    • RTT = Hello Retry Request
    • 2 méthodes : 0-RTT et 1-RTT

  • Problématique du 0-RTT
    • No PSK-ECDHE
    • Perfect Forward Secrecy not supported

  • Recommandaton :
    • Ne jamais utiliser 0-RTT mais 1-RTT
    • ou alors pour des requêtes GET sans paramètres

  • Pour plus d'informations :
    • https://blog.cloudflare.com/introducing-0-RTT/
    • TLS tris on GitHub

GET request with 0-RTT : les paramètres ne sont pas protégés par le Perfect Forward Secrecy

Subscribers geolocalisation and tracking using 4G VoLTE enabled Android phone

Par O. LE MOAL, P. VENTUZELO et T. COUDRAY (P1 Security)


  • VoLTE (Voice over LTE)
    • Based on VoIP Protocol (SIP, ...)
    • Peut être utilisé pour faire de la téléphonie par le Wifi et peut également supporter la vidéo

  • Attaque: Caller ID spoofing
    • Installation d'un logiciel pour forger des paquets 4G sur un téléphoné android rooté
    • L'attaquant forge un packet d'initialisation de communication avec un faux call id number
    • Modifcation des messages de signaling (non facturé et probablement non enregistré dans les écoutes téléphonique)
    • Il est également possible de modifier des headers "extra" (hors RFC) et d'ajouter des headers

  • Impact: récupération d'informations présentes dans le message de réponse
    • Modèle de téléphoné, build number et IMEI du destinataire du call
    • Utran-cell-id = géolocalisation du destinataire avec CallFinder qui recense les bornes des opérateurs

Et le téléphone du destinataire ne sonnera pas, il n'en saura rien...

DroneJack: kiss your drones goodbye !

par G. FOURNIER, P. COTRET, P. AUDREN DE KERDREL, V. VIET TRIEM TRONG (Centrale Supélec)


  • Solution existante : drone watch (brouilleur), drone defender (pistoler brouilleur), aigles, SkiJack (Attaque informatique)

  • Drone jack :
    • protection continue contre un nuage de drones
    • Sondes de détection des drones basé sur des Raspberry Pi

  • Fonctionnement
    • Phase de détection : le drone étant un point d'accès, détection des nouveaux point d'accès wifi (adresse MAC et autres critères)
    • Phase de tracking : analyse des trames "beacon" qui déterminent la distance en fonction de la puissance du signal
    • Phase de prise de controle: déconnection du pilote puis le drone se met en vol stationnaire puis prise de controle si réseau wifi ouvert

  • Finalité : récupération de la carte de vol pour identifier le point de décolage du drone et donc trouver le pilote qui se trouve généralement à ce point de décolage

Day 3 - Afternoon

Brosse à dent connectée

Par Axelle APVRILLE (Fortinet)


  • Brosse à dent
    • Non disponible pour le grand public
    • Commercialisée par une assurance dentaire souscrite par l'employeur
    • Une application mobile permet de jouer en se brossant les dent (Etoile, coeur, ...)

  • Attaque de la brosse à dent: se connecter à la brosse à dent à partir d'un ordinateur
    • BLE sans appairage
    • Information transférées : Vitesse moteur, niveau batterie, heure, ...
    • Possibilité de piloter la brosse à dent à distance : marche/arrêt, vitesse moteur, ...

  • Attaque de l'application mobile : simuler une brosse à dent à partir d'un ordinateur
    • Emission de trame BLE pour se faire passer pour une fausse brosse à dent aurpès de l'application mobile
    • Certains messages sont chiffrés (en AES CBC) et la clé se trouve en dur dans le code de l'application mobile
    • Application mobile utilise une API de l'assureur:
      • Modifier son score (coeur / etoile)
      • API en libre accès : Récupération des données des autres utilisateurs

Retour technique de l'incident

Par Anonyme (ANSSI)


Phase 1 : Compromission (Février 2015)

  • Compromission en 3 jours
  • Via un réseau VPN anonyme "IP Vanish"
  • Connexion au service RDP utilisé par les journalistes pour transmettre les vidéos, images et articles
  • Connexion avec le compte d'un prestataire sur l'Active Directory pour créer un compte administrateur de domaine
  • Installation du malware "ConnectBack" pour maintenir ses accés en cas de désactivation du RDP
  • Connexion au serveur de mails (Exchange) pour trouver des informations (@ip et mdp, ....)
  • Prise de controle du matériel audiovisuel: multiplexeur, encodeur, ... (Matériel Cisco avec mot de passe par défaut)

Retour technique de l'incident

Par Anonyme (ANSSI)


Phase 3 : Analyse

  • Récupération du mot de passe de l'administrateur de domaine frauduleux (avec Mimikatz)
  • Connection sur les sessions ouvertes avec "ConnnectBack.dll" pour analyse des actions effectuées
  • Collecte de données :

    • Flashback: forensic mémoire sur l'ensemble du parc
    • Orc2DB: centralisation des données permettant à tout les enquêteurs d'analyser les données

Retour technique de l'incident

Par Anonyme (ANSSI)


Phase 4 : Remediation

  • Mise en place d'un accès internet filtré (5 PC)
  • Mise en place d'un système de diffusion en boucle de vidéo déjà enregistrée
  • Création d'un environnement de travail minimaliste
  • Définition d'un réseau cloisonné : 50 VRF avec firewall
  • Installation d'un bastion RDP pour les prestataires
  • Durcissement de l'Active Directory: 3 zones (rouge, jaune et verte)
  • Bascule de l'Active Directory compromis vers l'Active Directory sécurisé sans interruption de service
